2 Layer 1 — Agent Files

📄

What is an Agent?

A single Python file responsible for one state or UT. It knows which districts to cover and how to fetch + parse weather data for each one.

📌

How many exist?

36 agents — one per state/UT. 28 state agents + 8 union territory agents. Each agent is completely independent.

🔌

Public Interface

Every agent exposes the same 3 functions: fetch(), get_summary(), get_all_summaries() — making them interchangeable.

Agent File Structure (e.g. agent_kerala.py)

Python

# ── 3. Raw API call ───────────────────────────────────
def fetch(city: str) -> dict:
    url = f"https://wttr.in/{city}?format=j1"
    response = requests.get(url, timeout=15)
    response.raise_for_status()
    return response.json()          # raw JSON from wttr.in

# ── 4. Parse one city ─────────────────────────────────
def get_summary(city: str) -> dict:
    data    = fetch(city)
    current = data["current_condition"][0]
    area    = data["nearest_area"][0]
    return {
        "city":          city,
        "temperature_c": current["temp_C"],
        "humidity":      current["humidity"],
        "description":   current["weatherDesc"][0]["value"],
        "wind_kmph":     current["windspeedKmph"],
        "uv_index":      current["uvIndex"],
        "forecast":      [...],  # 3-day forecast list
    }

# ── 5. Loop all districts ─────────────────────────────
def get_all_summaries() -> list:
    results = []
    for city in DISTRICTS:
        try:
            results.append(get_summary(city))  # success
        except Exception as exc:
            results.append({"city": city, "error": str(exc)})  # isolated failure
    return results

3 Layer 2 — Orchestrator

ⓘ

orchestrator.py is the central manager. It auto-discovers all 36 agents, runs them in parallel using a thread pool, aggregates the results, and outputs JSON + HTML. You never need to edit it when adding a new agent — it detects new agent_*.py files automatically.

How Agent Discovery Works

Python 
def _discover_agents():
    agents = {}
    # Scans for every file matching agent_*.py
    for path in sorted(BASE_DIR.glob("agent_*.py")):
        mod_name = f"india_weather.{path.stem}"
        mod = importlib.import_module(mod_name)  # dynamic import
        agents[mod.STATE] = mod                  # "Kerala" → module
    return agents

AGENTS = _discover_agents()   # runs once at startup → 36 agents

How Parallel Execution Works

Python 
def fetch_states(state_names, workers=8):
    # ThreadPoolExecutor runs 8 states simultaneously
    with ThreadPoolExecutor(max_workers=workers) as pool:
        futures = {
            pool.submit(fetch_state, state): state   # each state → own thread
            for state in state_names
        }
        for future in as_completed(futures):   # result arrives as it finishes
            result = future.result()
            print(f"[{done}/{total}] {state} ✓ {result['ok']} districts")

4 Layer 3 — HTML Dashboard

ⓘ

No server needed. The HTML file runs entirely in the browser. JavaScript fetches weather data directly from wttr.in using the browser's native fetch() API. You can open it by double-clicking the file — no Python required.

⚡

AbortController

When you click a state button mid-load, all in-flight requests are instantly cancelled. The selected state's requests start immediately instead of waiting in queue.

💾

dataCache

Every loaded city is stored in memory as dataCache["Kozhikode"] = {...}. Switching back to a previously viewed state shows data instantly — zero network calls.

🔄

Per-State Loading

Clicking "Kerala" fires only 14 requests, not 780. Background loading continues for other states after the selected one is done.

⚠

Wrong Location Guard

Every card checks country !== "India". If wttr.in resolves the city to another country, the resolved location turns red with a ⚠ warning.

JavaScript Engine Inside the HTML

JavaScript (inside HTML) 
// ── Globals ──────────────────────────────────────────
const dataCache  = {};         // { "Kozhikode": {...}, "Thrissur": {...} }
let   abortCtrl  = null;       // active AbortController
let   activeFilter = "all";    // current selected state

// ── Fetch one city ────────────────────────────────────
async function fetchCity(city, cardEl, signal) {
    const url  = `https://wttr.in/${city}?format=j1`;
    const resp = await fetch(url, { cache: "no-store", signal });
    // cache:"no-store" → browser never serves stale data
    // signal          → linked to AbortController (cancellable)
    const data = parse(await resp.json(), city);
    dataCache[city] = data;    // store in memory cache
    renderCard(cardEl, data);
}

// ── Load one state ────────────────────────────────────
async function loadState(state, signal) {
    state.cities.forEach((city, idx) => {
        if (dataCache[city]) renderCard(card, dataCache[city]);  // instant from cache
        else toFetch.push({ city, card });                       // needs network
    });
    // Fetch in batches of 6, 80ms apart
    for (let i = 0; i < toFetch.length; i += 6) {
        if (signal.aborted) return;   // stop if filter changed
        await Promise.allSettled(batch.map(t => fetchCity(t.city, t.card, signal)));
        await new Promise(r => setTimeout(r, 80));
    }
}

// ── Filter click ──────────────────────────────────────
function filterState(id, btn) {
    if (abortCtrl) abortCtrl.abort();   // ← CANCEL all in-flight fetches
    abortCtrl = new AbortController();   // fresh controller for new requests

    // Show only selected section
    STATES.forEach(s => section.style.display = (s.id === id) ? "" : "none");

    // Scroll into view + load only this state
    scrollTo(target); loadState(STATES.find(s => s.id === id), abortCtrl.signal);
}

5 End-to-End Data Flow

  1. User opens the HTML file

    Browser loads india_weather_report.html from disk. No server, no backend process is needed. JavaScript starts immediately.

  2. buildLayout() — DOM is created

    JavaScript reads the embedded STATES array (28 states + 8 UTs) and creates all the section headers, state buttons, and empty card placeholders in the DOM.

  3. loadAllStates() — sequential state-by-state loading

    States load one by one: Andhra Pradesh (26 cities), then Arunachal Pradesh (27), etc. Each state fires up to 6 concurrent requests. The next state starts only after the current one finishes.

  4. fetch() → wttr.in API

    For each district, the browser sends: GET https://wttr.in/Kozhikode?format=j1 with cache: "no-store". wttr.in's servers query the nearest weather station and return a JSON with current conditions + 3-day forecast.

  5. parse() — normalize the response

    Raw JSON is stripped to only the fields we need: temperature, humidity, wind, UV, visibility, pressure, cloud cover, observation time, and 3-day forecast. Also checks if country = India (wrong-location guard).

  6. dataCache[city] = data — stored in memory

    The parsed result is saved so if you switch away and come back to the same state, those cards render instantly without any network request.

  7. renderCard() — painted to the screen

    Card shows: city name, resolved location, observation time, temperature, weather icon, description, metrics grid (7 values), and a 3-day mini forecast strip.

  8. Auto-refresh every 10 minutes

    A countdown timer fires triggerRefresh() which clears the cache for the current view and re-fetches all data. The 10-minute interval matches wttr.in's own data update frequency.

6 wttr.in API — What We Get

⚠

Important: wttr.in is a free, public API with no authentication. It resolves city names by geocoding — if a name is ambiguous, it may return data for the wrong location. This is why we use "Banda Uttar Pradesh" instead of just "Banda" (which resolved to Argentina!).

Request Format

HTTP GET 
https://wttr.in/{city}?format=j1

Examples:
  https://wttr.in/Kozhikode?format=j1
  https://wttr.in/New%20Delhi?format=j1
  https://wttr.in/Banda%20Uttar%20Pradesh?format=j1   ← disambiguated
